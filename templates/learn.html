Guía de Logs para Blue Team

Todo lo que necesitas saber sobre logs de seguridad, desde cero hasta nivel analista SOC.

📋

¿Qué es un log?

Un log es un registro cronológico de eventos que ocurren en un sistema. Cada vez que alguien inicia sesión, accede a un archivo, ejecuta un proceso o establece una conexión de red, el sistema genera un evento que queda registrado.

Para un analista Blue Team, los logs son la fuente principal de evidencia. Son el equivalente digital de las huellas que deja un atacante al moverse por una red.

💡
Analogía: Imagina un hotel. Cada vez que alguien entra por la puerta, sube al ascensor, abre una habitación o usa la wifi, el sistema del hotel lo registra. Los logs son ese registro. Si alguien roba algo, el investigador revisa los registros para saber quién estuvo, cuándo y dónde.
⏱️ Timestamp Cuándo ocurrió el evento exactamente
🖥️ Origen Qué sistema o aplicación lo generó
📝 Descripción Qué acción se realizó exactamente
👤 Usuario Quién realizó la acción
🌐 IP / Host Desde dónde se realizó
🔢 Event ID Código único que identifica el tipo de evento
🪟 Windows Event Logs

Registros nativos de Windows. Cubren autenticación, procesos, acceso a archivos y cambios de configuración. Son la fuente principal en entornos empresariales Windows.

Security.evtx System.evtx Application.evtx
🔬 Sysmon

Herramienta de Microsoft que amplía los logs nativos. Registra creación de procesos con línea de comandos completa, conexiones de red, modificaciones de registro y mucho más.

ProcessCreate NetworkConnect RegistryEvent
🌐 Logs de Red

Registros generados por firewalls, proxies y dispositivos de red. Muestran qué IPs se comunicaron, en qué puertos y cuántos datos transfirieron.

Firewall logs Proxy logs DNS logs
🐧 Linux Syslog

Sistema de logging estándar en Linux. Registra eventos del sistema operativo, servicios y aplicaciones. Clave en servidores web, bases de datos y infraestructura cloud.

/var/log/syslog /var/log/auth.log /var/log/secure
🌍 Web Server Logs

Registros de servidores web (Apache, Nginx, IIS). Cada petición HTTP queda registrada con URL, código de respuesta, IP origen y User-Agent. Detectan SQLi, XSS y web shells.

access.log error.log IIS logs
🛡️ EDR / AV Logs

Logs generados por soluciones EDR (CrowdStrike, SentinelOne, Wazuh) y antivirus. Alertas de detección, comportamientos bloqueados y respuestas automáticas.

Wazuh alerts Detections Quarantine events

Event IDs críticos para Blue Team

Cada evento en Windows tiene un ID único. Estos son los más importantes que debes conocer como analista SOC:

Event ID Descripción Relevancia
4624 Inicio de sesión exitoso Monitorizar
4625 Fallo de autenticación Alta prioridad
4688 Creación de nuevo proceso Alta prioridad
4698 Tarea programada creada Persistencia
4720 Cuenta de usuario creada Alta prioridad
4732 Usuario añadido a grupo privilegiado Alta prioridad
4776 Validación de credenciales NTLM Lateral movement
7045 Nuevo servicio instalado Persistencia
⚡
Truco pro: En Kibana o Splunk, filtra siempre por Event ID 4688 (creación de procesos) para detectar ejecución de malware. Un proceso como powershell.exe siendo hijo de outlook.exe es una señal de alarma inmediata.
🔍 Elastic Stack (ELK)

Elasticsearch + Kibana es el SIEM open-source más usado en el mundo. Los logs se indexan en Elasticsearch y se visualizan en Kibana con dashboards y búsquedas KQL.

Ejemplo KQL — Buscar proceso sospechoso: process.name: "powershell.exe" AND process.parent.name: "outlook.exe"
📊 Splunk

El SIEM más demandado en ofertas de trabajo SOC. Usa SPL (Splunk Processing Language) para buscar y correlacionar eventos. Muy potente para detección de anomalías.

Ejemplo SPL — Fallos de login por usuario: index=security EventCode=4625 | stats count by Account_Name | where count > 10
🛡️ Wazuh

SIEM + EDR open-source. Combina análisis de logs con detección de intrusiones en endpoints. Muy usado en PYMEs y gobiernos. Incluye compliance con PCI-DSS y GDPR.

Regla Wazuh — Detectar conexión C2: rule id="100001" — dst_ip not in whitelist AND dst_port=443 AND process=svchost

Indicadores de Compromiso (IOCs) en logs

Un IOC es cualquier artefacto observado en logs que indica con alta probabilidad que un sistema ha sido comprometido. Saber identificarlos es la habilidad central de un analista SOC.

🔴 IOCs de Alta Criticidad
  • Proceso ejecutado desde C:\Users\Public\ o C:\Temp\
  • PowerShell con parámetro -enc (base64 ofuscado)
  • Proceso padre inusual — outlook.exe → cmd.exe
  • Conexión saliente a IP Tor o conocida C2
  • Modificación de clave Run en registro
  • Nuevo servicio instalado fuera de horario
🟡 IOCs de Media Criticidad
  • Múltiples fallos de login seguidos de éxito
  • Acceso a archivos fuera de horario laboral
  • User-Agent obsoleto o inusual en tráfico web
  • Escaneo de puertos desde IP interna
  • Transferencia inusualmente grande de datos
  • DNS query a dominio recién registrado (DGA)
🔵 Falsos Positivos Comunes
  • Administrador ejecutando scripts de mantenimiento
  • Escáner de vulnerabilidades interno autorizado
  • Backup nocturno transfiriendo datos masivamente
  • Usuario olvidando contraseña repetidamente
  • Software de monitorización generando conexiones
  • Actualización automática de software
